# Спецификация: тонкий патч-образ Тундука (флавор A)

Дата: 2026-06-22
Статус: реализуется (E2E по запросу пользователя)

## Почему это вообще понадобилось

Прежнее утверждение репозитория - "Тундук = байт-в-байт ванильный NIIS, своего образа
не нужно" - оказалось НЕВЕРНЫМ. Оно было основано на сравнении только метаданных
индекса пакетов (Maintainer, версия, git-хэш - они совпадают). Содержимое .deb не
вскрывалось.

Проверка по факту (вскрытие пакетов + сравнение с официальным образом
niis/xroad-security-server-sidecar:7.4.2):

- В Тундуковском `xroad-proxy` пакете (внутри `proxy-1.0.jar`) есть РОВНО 3 класса,
  которых НЕТ в официальном образе:
  - `ee/ria/xroad/common/certificateprofile/impl/KgSkKlass3CertificateProfileInfoProvider.class`
  - тот же класс + inner `KgSkAuthCertificateProfileInfo`
  - тот же класс + inner `KgSkSignCertificateProfileInfo`
- Больше отличий нет: 0 удалённых классов, `proxy-ui-api-1.0.jar` идентичен по составу.
- Это провайдер профиля сертификата для УЦ Кыргызстана (ГУ "Кызмат", KgSk = KG + SK Klass3).
  Без него сервер не может сгенерировать CSR в формате, который примет УЦ - на это
  напоролись при создании ключа AUTH.

Вывод: "тундуковость" - это НЕ только якорь в рантайме, как мы писали. Есть ещё патч в
бинарнике (один Java-класс). Официальный образ NIIS для Тундука в проде не годится как есть.

## Решение: тонкий патч-образ поверх официального

Подход A (выбран пользователем): `FROM niis/xroad-security-server-sidecar:7.4.2` и наложить
сверху Тундуковские jar'ы из официального репозитория deb.tunduk.kg. Сохраняем весь
удобный sidecar-обвес NIIS (supervisord, авто-создание БД, env), меняем ровно то, что
отличается у Тундука.

Почему overlay jar'ов, а не apt install:
- Пакеты `xroad-proxy` и `xroad-proxy-ui-api` имеют `Architecture: all` (чистая Java) -
  их jar'ы независимы от ОС, безопасно класть с jammy-пакета на focal-образ.
- Официальный образ собран на ubuntu20.04 (focal), пакеты Тундука - ubuntu22.04 (jammy).
  `apt install` jammy-пакетов на focal потянул бы native-зависимости (xroad-base, signer,
  confclient - Architecture: amd64) и мог сломать образ. Overlay только Java-jar этого избегает.
- Патч аддитивный (3 класса добавлены, ничего не убрано) - замена двух jar воспроизводит
  поведение Тундука без риска для остального образа.

## Образ (image/Dockerfile)

```
FROM niis/xroad-security-server-sidecar:7.4.2
ARG TUNDUK_REPO=https://deb.tunduk.kg/ubuntu22.04-7.4.2
# скачать два пакета по точному имени файла (несёт версию+git-хэш),
# сверить sha256, распаковать dpkg-deb, заменить на месте:
#   /usr/share/xroad/jlib/proxy-1.0.jar
#   /usr/share/xroad/jlib/proxy-ui-api-1.0.jar
# (симлинки proxy.jar / proxy-ui-api.jar указывают на эти файлы)
```

Пиннинг (воспроизводимость + ловит тихое изменение апстрима):
- Точные имена файлов: `xroad-proxy_7.4.2-1.gita30be58.ubuntu22.04_all.deb`,
  `xroad-proxy-ui-api_7.4.2-1.gita30be58.ubuntu22.04_all.deb`.
- sha256 (проверено на скачанных файлах):
  - proxy:        `70a7f30ad6ff09e3813ae64c81573fda18fed266a2d15df8c0c5392077ef3b3c`
  - proxy-ui-api: `806706aaab34bec3e60f8fed1e0b8acadfedb8c32d8b7b1fa74aa846078853fe`
- В базовом образе есть curl, dpkg-deb, tar - достаточно для скачивания и распаковки.

Тег собранного образа: `tunduk-security-server:7.4.2` (локально для compose; для k8s
пушится в реестр пользователя).

## Как флаворы потребляют образ

- docker-compose: в сервис `security-server` добавляется `build:` (контекст `../image`)
  + `image: tunduk-security-server:7.4.2`. `docker compose up --build` собирает патч-образ
  локально. Внешний реестр для одной виртуалки не нужен. Путь остаётся одной командой.
- Helm: Kubernetes не умеет собирать образ. `image.repository` по умолчанию указывает на
  `tunduk-security-server` (тег из appVersion). README чарта документирует: собрать образ и
  запушить в свой реестр (например ghcr.io/goodwokdev/tunduk-security-server), затем
  переопределить `image.repository`. Конкретный реестр в дефолт не хардкодим.

## Проверка (доказуемо, а не на словах)

- Build-тест: `docker build` образа, затем grep по proxy.jar внутри собранного образа -
  класс `KgSkKlass3CertificateProfileInfoProvider` ДОЛЖЕН присутствовать (тот же grep,
  что доказал его отсутствие в стоковом NIIS).
- `docker compose config` валиден, `helm lint` чистый, `helm template` рендерится.

Граница проверки (честно): локально доказываем, что класс KgSk появился в образе. Полный
E2E (что CSR реально принимается УЦ "Кызмат") требует якоря + зарегистрированной личности
и проверяется на стороне Dena. Это отмечается в документации.

## Исправление неверного утверждения (обязательно, оно опубликовано)

Репозиторий уже публичный (github.com/goodwokdev/tunduk-security-server). Утверждение про
"ванильный NIIS, своего образа не надо" - фактическая ошибка и должна быть исправлена в:
- README.md (раздел "Почему нет своего образа" -> переписать)
- docs/guide-for-newcomers.md (раздел "почему не пришлось собирать свой образ")
- комментарии docker-compose.yml
- helm-charts/.../README.md (раздел "Почему нет своего образа")
- файл памяти проекта

Новая формулировка: метаданные пакетов совпадают с NIIS, НО Тундук добавляет KG-класс
профиля сертификата (KgSkKlass3) в proxy.jar; поэтому нужен тонкий патч-образ поверх
официального. Это лучшая практика для данного случая: минимальный, воспроизводимый,
прозрачный в git патч.

## Что осознанно НЕ делаем (YAGNI)

- Не пересобираем X-Road из исходников и не собираем образ Тундука с нуля.
- Не тянем все пакеты Тундука через apt (риск focal/jammy, не нужно - патч в 2 jar).
- Не компилируем KgSk-класс сами (исходник может быть не опубликован; overlay готового
  jar из официального репо Тундука проще и надёжнее).
- Не хардкодим конкретный контейнерный реестр для Helm.
