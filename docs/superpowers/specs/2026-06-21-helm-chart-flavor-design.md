# Спецификация: Helm-чарт как второй флавор развёртывания Тундук SS

Дата: 2026-06-21
Статус: на ревью

## Контекст и цель

В репозитории уже есть рабочий флавор развёртывания на docker-compose (одна виртуалка,
официальный образ niis/xroad-security-server-sidecar:7.4.2, отдельный PostgreSQL 12,
безопасные по умолчанию порты, постоянные тома). Он остаётся опорным, эталонным флавором:
по нему проще всего понять, как устроен сервер безопасности, и поднять его на одной машине.

Цель этой работы - добавить второй флавор: Helm-чарт для Kubernetes. Чарт не заменяет
compose, а становится альтернативным способом развернуть то же самое в кластере.

Важная честная оговорка: два флавора разделяют не файлы, а контракт. YAML compose и шаблоны
Helm не DRY-ятся друг в друга. Одинаковым у них остаётся именно контракт развёртывания:
тег образа, имена переменных окружения, раскладка томов, политика портов. Чарт повторяет
семантику compose, чтобы всё, что ты понял по одному флавору, переносилось на другой.

## Ключевые ограничения (проверены в этой сессии)

- Образ niis/xroad-security-server-sidecar:7.4.2 несёт внутри PostgreSQL 12 (проверено по
  реальному образу: pg_ctlcluster 12). Бэкап и восстановление ломаются между мажорными
  версиями PostgreSQL. Поэтому внешняя БД обязана быть именно версии 12. Документация master
  говорит "16", но это для более новой версии X-Road, к 7.4.2 не относится.
- PostgreSQL 12 уже снят с поддержки (EOL), официальных патчей по нему нет. Это осознанный
  компромисс ради совместимости с образом. Снижаем риск тем, что БД живёт внутри кластера и
  не торчит наружу.
- У сервера безопасности уникальная криптографическая личность: свои ключи AUTH/SIGN и
  зарегистрированный код сервера (server code). Нельзя наивно поднять replicas: 3 - это будет
  одна и та же личность с нескольких адресов, и Тундук это отклонит как конфликт. Поэтому
  одиночный инстанс жёстко ограничен одной репликой.
- Тундук - это ванильный X-Road (пакеты deb.tunduk.kg байт-в-байт совпадают с NIIS).
  "Тундуковским" сервер делает только глобальный якорь конфигурации, который грузится в рантайме.

## Решения (зафиксированы с пользователем)

- Форма чарта: одиночный, production-faithful. Один под = одна личность сервера, точное
  зеркало compose-флавора. Не HA-кластер.
- База данных: встроенный postgres:12 StatefulSet с PVC. Дополнительно - переключатель на
  внешнюю БД (по умолчанию выключен), это почти бесплатно в шаблоне и полезно для prod.
- Экспозиция (выбрано как разумный дефолт, кластер сейчас недоступен для проверки): все
  Service по умолчанию ClusterIP; порты 5500/5577 можно переключить на LoadBalancer/NodePort
  через values; админ-панель достаётся через kubectl port-forward (k8s-эквивалент привязки к
  loopback в compose).
- Язык документации: русский.

## Архитектура

Чарт разворачивает один сервер безопасности и (по умолчанию) одну БД PostgreSQL 12 в кластере.

```
namespace
  StatefulSet security-server (replicas: 1, жёстко)
    volumeClaimTemplates:
      etc-xroad   -> /etc/xroad        (ключи, сертификаты, якорь)
      var-xroad   -> /var/lib/xroad    (архив журнала сообщений, 3 года)
    env <- Secret (PIN, admin, пароль БД)
    probes: readiness/startup на :4000
  StatefulSet db (postgres:12, если postgresql.embedded=true)
    volumeClaimTemplate: pgdata -> /var/lib/postgresql/data
  Services:
    ss-admin   ClusterIP  4000        (port-forward, опц. ingress)
    ss-is      ClusterIP  8080,8443   (доступ информационных систем)
    ss-xroad   ClusterIP  5500,5577   (обмен между серверами; опц. LB/NodePort)
    db         Headless   5432
  Secret: XROAD_TOKEN_PIN, XROAD_ADMIN_USER/PASSWORD, пароль БД
```

## Компоненты

### Сервер безопасности (StatefulSet)

- replicas: 1, жёстко. В values стоит guard и комментарий с объяснением (уникальная личность,
  нельзя наивно масштабировать). Если кто-то поставит больше - чарт должен это пресекать или
  явно предупреждать через NOTES/комментарий.
- volumeClaimTemplates на /etc/xroad и /var/lib/xroad. Потеря тома etc-xroad = перевыпуск
  сертификатов, поэтому он постоянный.
- readinessProbe на :4000 (это проверенно-надёжный сигнал готовности, работает и до, и после
  регистрации). startupProbe с щедрым окном на запуск JVM-служб и первичную инициализацию БД.
- Образ и тег - значения по умолчанию из values, тег 7.4.2 (full, не slim: slim не умеет
  журналирование сообщений, а оно по закону Тундука обязательно).

### База данных (StatefulSet)

- postgres:12, один PVC через volumeClaimTemplate.
- Включается через postgresql.embedded (по умолчанию true).
- Гонку старта решаем структурно: startupProbe сервера ретраит, пока БД не ответит. Это
  k8s-аналог depends_on service_healthy из compose. Энтрипоинт образа сам создаёт нужные базы
  (serverconf, messagelog, op-monitor) и db.properties, подключаясь суперпользователем.

### Внешняя БД (опционально)

- Блок externalDatabase в values: host, port, password, existingSecret.
- Используется при postgresql.embedded=false. Major-версия обязана быть 12 - это явно
  написано в README и в комментарии values.

### Секреты и конфигурация

- Secret хранит XROAD_TOKEN_PIN, XROAD_ADMIN_USER, XROAD_ADMIN_PASSWORD и пароль БД.
- Два режима: создаётся чартом из values, либо берётся готовый через existingSecret
  (рекомендуется для prod). Имена переменных окружения совпадают с compose-флавором.
- Якорь конфигурации не запекается в образ. Грузится либо через UI по SETUP.md, либо
  монтируется из предоставленного пользователем ConfigMap/Secret.

### Экспозиция (Services)

- ss-admin (4000) и ss-is (8080/8443): ClusterIP. Доступ к панели по kubectl port-forward.
  Опциональный блок ingress для админ-панели (по умолчанию выключен).
- ss-xroad (5500/5577): ClusterIP по умолчанию, переключается на LoadBalancer/NodePort через
  service.xroad.type. Это единственные порты, которым положено покидать кластер.
- db: headless service для StatefulSet.

Политика портов повторяет принцип compose: наружу осознанно выходят только 5500/5577 (там
mTLS), всё остальное по умолчанию внутри.

## Документация

- README чарта на русском: справочник по values, явно вынесенное ограничение PG12, заметка

- Короткий раздел про два флавора в корневом README.

## Что осознанно НЕ делаем (YAGNI)

- HA / кластерную топологию (primary + secondary, общая БД, балансировщик с SSL passthrough).
  Это отдельный большой объём, не для этого чарта.
- Автоматизацию регистрации (якорь, ключи, сертификаты) - она завязана на внешних людей и УЦ,
  остаётся ручной по SETUP.md.
- Бэкап-оператор для k8s - на этом этапе бэкап описан как процедура, не как контроллер.
