# Сервер безопасности Тундук - руководство по настройке и регистрации

Путь от чистой виртуалки до **зарегистрированного** сервера безопасности Тундук с
**зарегистрированной подсистемой**, готовой к обмену данными в СМЭВ.

Docker-стек (см. [README.md](README.md)) даёт запущенный, но *ненастроенный* сервер.
Доведение его до состояния `registered` - это последовательность ниже. Часть шагов
зависит от **внешних участников, у которых есть свои сроки** - начинай раздел 0
сразу, параллельно со всем остальным.

---

## 0. Внешние зависимости - начинай с них (они блокируют всё остальное)

Это настоящие длинные хвосты. Запускай их в первый же день:

1. **Глобальный якорь конфигурации** - запроси файл якоря у администраторов Тундука
   / СМЭВ. Хэш якоря нужно будет сверить с опубликованным значением.
2. **Реквизиты участника** - получи у администраторов Тундука:
   - **Класс** участника: `GOV` (государственный) или `COM` (коммерческий).
   - **Код** участника (код твоей организации).
3. **Договор с УЦ ГУ "Кызмат"** - заключи договор на PKI на
   <https://gukyzmat.gov.kg/pki>. Пакет документов для юридических лиц:
   - Соглашение о присоединении к Регламенту УЦ ГУ "Кызмат"
   - Доверенность пользователя УЦ (для представителей юр. лиц)
   - Копия паспорта пользователя УЦ
   - Копия приказа о назначении уполномоченного лица
   - Копия приказа о назначении руководителя организации
   - Копия свидетельства о госрегистрации/перерегистрации юр. лица
   - Сопроводительное письмо в адрес ГУ "Кызмат"

Заранее ты также выбираешь:
- **Код сервера безопасности (server code)** - только латиница, должен содержать имя
  организации, уникален в пределах инстанса Тундука. **Кириллица здесь = подписанный
  сертификат отклонят.**
- **PIN программного токена (SoftToken PIN)** - минимум 15 символов, верхний и нижний
  регистр, цифры, знаки. Храни надёжно; потеря PIN заставит перевыпускать все
  сертификаты. Должен совпадать с `XROAD_TOKEN_PIN`.

## 1. Требования к хосту

- Виртуалка физически в Кыргызстане, установлены Docker Engine + Compose v2.
- Выделена под X-Road (владеет опубликованными портами - другого веб-сервера нет).
- Минимум 4 GB RAM, 100 GB свободного диска.

### Фаервол / порты

| Напр. | Порт(ы) | Назначение |
|---|---|---|
| вх | `5500`, `5577` /tcp | обмен сообщениями + OCSP между серверами безопасности (от других SS) |
| вх | `4000` /tcp | админ-панель (держать в админ-подсети / за SSH-туннелем) |
| вх | `8080`, `8443` /tcp | точки доступа информационных систем (только от твоей внутренней ИС) |
| вх | `80` /tcp | ACME (только если используешь сертификаты через ACME) |
| исх | `5500`, `5577` /tcp | к другим серверам безопасности |
| исх | `80`, `443`, `4001` /tcp | центральный сервер + глобальная конфигурация |
| исх | `62301`, `62302` /tcp | OCSP УЦ + служба меток времени |
| исх | `123` /udp | NTP |

Ограничь `4000`/`8080`/`8443` доверенными источниками. `5500`/`5577` должны быть
доступны из более широкой сети X-Road.

## 2. Поднять стек

```bash
cd docker-compose
cp .env.example .env
# Отредактируй .env: XROAD_TOKEN_PIN (== твой SoftToken PIN), учётка панели, пароль БД.
docker compose up -d
docker compose ps        # дождись, пока оба сервиса -> healthy
docker compose logs -f security-server
```

На первом старте энтрипоинт создаёт `/etc/xroad/db.properties`, подключается к
Postgres суперпользователем и автоматически создаёт базы и роли `serverconf` /
`messagelog` / `op-monitor`. Настраивать БД руками не нужно.

Прокинь туннель к панели и залогинься (TLS самоподписанный):

```bash
ssh -L 4000:127.0.0.1:4000 <vm-host>
# https://localhost:4000  - учётные данные = XROAD_ADMIN_USER / XROAD_ADMIN_PASSWORD
```

## 3. Первоначальная настройка (админ-панель)

1. **Загрузи якорь** (раздел 0.1). Сверь показанный хэш с опубликованным значением.
2. Введи, когда попросит:
   - **Класс** участника (`GOV`/`COM`) и **код** участника (раздел 0.2).
   - **Код сервера безопасности** (латиница, раздел 0).
   - **SoftToken PIN** - должен совпадать с `XROAD_TOKEN_PIN` в `.env`.
3. Панель покажет имя владельца, зарегистрированное на центральном сервере - проверь,
   что оно верное.

---

## 4. Служба меток времени (timestamping)

`SETTINGS > System Parameters > Timestamping Services > ADD` > выбери TSP Тундука.
(Обязательно; сервер безопасности подписывает журнал сообщений против неё.)

---

## 5. Ключи и запросы на подпись сертификатов

В разделе **Keys and Certificates** открой SoftToken (введи PIN), нажми **Add key** дважды:

| Ключ | Label | Что делать |
|---|---|---|
| AUTH | authentication | `Generate CSR`, **Country Code (C) = KG** > `Done` > скачать `auth_csr_*.der` |
| SIGN | signing | `Generate CSR`, **Country Code (C) = KG** > `Done` > скачать `sign_csr_*.der` |

Отправь **оба** файла `.der` на почту удостоверяющего центра: **kuc@infocom.kg**.
(Для этого нужен заключённый договор с ГУ "Кызмат" из раздела 0.3.)

Дождись, пока УЦ вернёт два подписанных сертификата.

---

## 6. Импорт, регистрация, активация

1. **Import cert** - импортируй **оба** возвращённых сертификата (auth + sign) на
   странице Keys and Certificates.
2. Рядом с сертификатом **auth** нажми **Register**; введи DNS-имя сервера или внешний
   IP. Статус: `saved` > `registration in progress`.
3. Подожди ~10-15 минут одобрения Центром электронного взаимодействия Тундука.
4. После одобрения нажми на сертификат > **Activate** (активируй и на своей стороне).

> Статус OCSP застрял на `Unknown` вместо `Good`?
> `docker compose restart security-server`, либо посмотри в Diagnostics время
> следующего обновления OCSP (по умолчанию интервал ~48 минут без ручных запросов).

Сервер безопасности зарегистрирован.

---

## 7. Регистрация подсистемы (обмен данными СМЭВ)

1. Выбери **Subsystem Code** - только ASCII / латиница.
2. **Clients > Add subsystem** > введи Subsystem Code > **OK**.
3. Нажми **Register**, чтобы отправить запрос.
4. **Уведоми ОАО "Тундук"**, что ты зарегистрировал подсистему (этот шаг ручной -
   автоматически они её не увидят).
5. Статус: `registration in progress` > ~10 минут > `registered`.

Протокол подключения по умолчанию **HTTPS**. Чтобы использовать HTTP, переключи это в
свойствах подсистемы. Про клиентские HTTPS-сертификаты см. статью Тундука
"Настройка HTTPS-сертификатов подсистемы".

Переносишь существующую подсистему со старого сервера 6.21.1? Введи прежнее имя
подсистемы, уведоми администратора Тундука о переносе и пересоздай сервисы вручную.

## 8. Бэкап и восстановление

**Бэкап** (по расписанию через cron - копию уноси с машины, для 3-летнего хранения):

```bash
backup/backup.sh
# создаёт backup/out/{db,etc-xroad,var-lib-xroad}-<stamp>.{sql.gz,tar.gz}
```

**Восстановление** на чистый стек (та же версия образа, Postgres 12). Команды
запускаются из каталога `docker-compose/`, бэкапы лежат в `../backup/out/`:

```bash
cd docker-compose
docker compose up -d db                       # запустить только БД
gunzip -c ../backup/out/db-<stamp>.sql.gz | \
  docker compose exec -T -e PGPASSWORD="$XROAD_DB_PWD" db psql -U postgres
docker compose up -d                          # запустить SS
# восстанови /etc/xroad и /var/lib/xroad в их тома, например:
docker compose exec -T security-server tar -C / -xzf - < ../backup/out/etc-xroad-<stamp>.tar.gz
docker compose exec -T security-server tar -C / -xzf - < ../backup/out/var-lib-xroad-<stamp>.tar.gz
docker compose restart security-server
```

> Восстановление требует **того же мажора PostgreSQL (12)** - восстановление между
> версиями не работает.
